<h2 id="executive-summary">Executive Summary</h2>
<h3 id="overview">Overview</h3>
<p>This comprehensive guide explores SQL Server table partitioning, a powerful technique for managing large tables by dividing them into smaller, more manageable pieces. Partitioning improves query performance, simplifies maintenance operations, and enables efficient data archiving for tables with millions or billions of rows.</p>
<h3 id="key-takeaways">Key Takeaways</h3>
<ul>
<li>
<p><strong>Horizontal Partitioning:</strong> Divide large tables into smaller partitions based on a partition key, improving manageability and performance</p>
</li>
<li>
<p><strong>Three-Step Process:</strong> Partitioning requires a partition function (defines ranges), partition scheme (maps to filegroups), and partitioned table</p>
</li>
<li>
<p><strong>Partition Elimination:</strong> Queries automatically scan only relevant partitions, dramatically reducing I/O and improving performance</p>
</li>
<li>
<p><strong>Sliding Window:</strong> Efficiently add new partitions and remove old ones without impacting production queries</p>
</li>
<li>
<p><strong>Aligned Indexes:</strong> Keep indexes aligned with table partitions for optimal performance and maintenance</p>
</li>
<li>
<p><strong>Enterprise Feature:</strong> Table partitioning requires SQL Server Enterprise Edition (or Azure SQL Database)</p>
</li>
</ul>

<h2 id="introduction">Introduction</h2>
<p>Imagine managing a table with billions of rows representing years of transaction history. Simple queries take minutes to execute, backups run for hours, and index maintenance locks the table for extended periods. Your users complain about slow response times, and your maintenance windows keep expanding.</p>
<p>This is where table partitioning becomes essential. By dividing your massive table into smaller, manageable partitions, you can dramatically improve query performance, enable efficient data archiving, and perform maintenance on individual partitions without affecting the entire table.</p>
<p>SQL Server's table partitioning feature transforms how you manage large-scale data, making previously impossible operations practical and efficient.</p>

<h2 id="understanding-table-partitioning">Understanding Table Partitioning</h2>

<h3 id="why-partition-tables">Why Partition Tables?</h3>
<p><strong>Problem 1: Query Performance</strong>
- âŒ Without partitioning: Scan billions of rows even for recent data
- âœ… With partitioning: Scan only relevant partition (partition elimination)</p>
<p><strong>Problem 2: Maintenance Operations</strong>
- âŒ Without partitioning: Index rebuild locks entire table for hours
- âœ… With partitioning: Rebuild one partition at a time, minimal impact</p>
<p><strong>Problem 3: Data Archiving</strong>
- âŒ Without partitioning: DELETE millions of rows = slow, log intensive
- âœ… With partitioning: SWITCH partition out instantly, no logging</p>
<p><strong>Problem 4: Backup and Restore</strong>
- âŒ Without partitioning: All-or-nothing backup/restore
- âœ… With partitioning: Backup/restore specific partitions</p>
<h3 id="real-world-analogy">Real-World Analogy</h3>
<p>Think of table partitioning like organizing a <strong>library by year</strong>:</p>
<p><strong>Non-Partitioned (One Big Room):</strong>
- All books from 1900-2024 mixed together
- Finding a 2023 book requires searching the entire room
- Cleaning means closing the entire library
- Moving old books requires sorting through everything</p>
<p><strong>Partitioned (Separate Rooms by Decade):</strong>
- 1900s room, 1910s room, 1920s room... 2020s room
- Finding a 2023 book? Go directly to the 2020s room
- Clean one room while others remain open
- Archive 1900s books? Move the entire room at once</p>

<h2 id="partitioning-fundamentals">Partitioning Fundamentals</h2>
<h3 id="partition-function">Partition Function</h3>
<p>A <strong>partition function</strong> defines how data is divided into partitions using boundary values.</p>
<pre class="codehilite"><code class="language-sql">-- Example: Partition by year
CREATE PARTITION FUNCTION PF_SalesByYear (DATE)
AS RANGE RIGHT FOR VALUES
    ('2020-01-01', '2021-01-01', '2022-01-01', '2023-01-01');
</code></pre>

<p><strong>What this creates:</strong></p>
<pre class="codehilite"><code>Partition 1: &lt; 2020-01-01
Partition 2: &gt;= 2020-01-01 AND &lt; 2021-01-01
Partition 3: &gt;= 2021-01-01 AND &lt; 2022-01-01
Partition 4: &gt;= 2022-01-01 AND &lt; 2023-01-01
Partition 5: &gt;= 2023-01-01
</code></pre>

<h3 id="partition-scheme">Partition Scheme</h3>
<p>A <strong>partition scheme</strong> maps partitions to filegroups (storage locations).</p>
<pre class="codehilite"><code class="language-sql">-- Map partitions to filegroups
CREATE PARTITION SCHEME PS_SalesByYear
AS PARTITION PF_SalesByYear
TO (FG_2019, FG_2020, FG_2021, FG_2022, FG_2023);
</code></pre>

<h3 id="partitioned-table">Partitioned Table</h3>
<p>Create table using the partition scheme:</p>
<pre class="codehilite"><code class="language-sql">CREATE TABLE Sales (
    SaleID INT IDENTITY(1,1),
    SaleDate DATE NOT NULL,
    CustomerID INT,
    Amount DECIMAL(18,2),
    CONSTRAINT PK_Sales PRIMARY KEY (SaleID, SaleDate)
)
ON PS_SalesByYear(SaleDate);  -- Partition on SaleDate
</code></pre>

<h3 id="choosing-the-right-boundary">Choosing the Right Boundary</h3>
<p><strong>For Dates: Use RANGE RIGHT</strong></p>
<pre class="codehilite"><code class="language-sql">-- RANGE RIGHT for dates (recommended)
CREATE PARTITION FUNCTION PF_ByMonth (DATE)
AS RANGE RIGHT FOR VALUES
    ('2024-01-01', '2024-02-01', '2024-03-01');

-- Result:
-- Partition 1: &lt; 2024-01-01 (old data)
-- Partition 2: Jan 2024 (&gt;= 2024-01-01 AND &lt; 2024-02-01)
-- Partition 3: Feb 2024 (&gt;= 2024-02-01 AND &lt; 2024-03-01)
-- Partition 4: Mar+ 2024 (&gt;= 2024-03-01)
</code></pre>

<p><strong>Why RANGE RIGHT for dates?</strong>
- More intuitive: "January partition starts on 2024-01-01"
- Easier sliding window maintenance
- Aligns with business logic</p>

<h2 id="creating-partitioned-tables">Creating Partitioned Tables</h2>
<h3 id="step-by-step-implementation">Step-by-Step Implementation</h3>
<p><strong>Step 1: Create Filegroups (Optional but Recommended)</strong></p>
<pre class="codehilite"><code class="language-sql">-- Create filegroups for each partition
ALTER DATABASE SalesDB
ADD FILEGROUP FG_Sales_2020;

ALTER DATABASE SalesDB
ADD FILEGROUP FG_Sales_2021;

ALTER DATABASE SalesDB
ADD FILEGROUP FG_Sales_2022;

ALTER DATABASE SalesDB
ADD FILEGROUP FG_Sales_2023;

-- Add files to filegroups
ALTER DATABASE SalesDB
ADD FILE (
    NAME = Sales_2020,
    FILENAME = 'D:\Data\Sales_2020.ndf',
    SIZE = 100MB,
    FILEGROWTH = 50MB
) TO FILEGROUP FG_Sales_2020;

-- Repeat for other filegroups...
</code></pre>

<p><strong>Step 2: Create Partition Function</strong></p>
<pre class="codehilite"><code class="language-sql">CREATE PARTITION FUNCTION PF_SalesByYear (DATE)
AS RANGE RIGHT FOR VALUES
    ('2020-01-01', '2021-01-01', '2022-01-01', '2023-01-01');
</code></pre>

<p><strong>Step 3: Create Partition Scheme</strong></p>
<pre class="codehilite"><code class="language-sql">CREATE PARTITION SCHEME PS_SalesByYear
AS PARTITION PF_SalesByYear
TO (FG_Sales_2020, FG_Sales_2021, FG_Sales_2022,
    FG_Sales_2023, PRIMARY);
-- PRIMARY for future data beyond 2023
</code></pre>

<p><strong>Step 4: Create Partitioned Table</strong></p>
<pre class="codehilite"><code class="language-sql">CREATE TABLE Sales (
    SaleID BIGINT IDENTITY(1,1),
    SaleDate DATE NOT NULL,
    CustomerID INT NOT NULL,
    ProductID INT NOT NULL,
    Quantity INT NOT NULL,
    UnitPrice DECIMAL(18,2) NOT NULL,
    TotalAmount AS (Quantity * UnitPrice) PERSISTED,
    CONSTRAINT PK_Sales PRIMARY KEY (SaleID, SaleDate)
) ON PS_SalesByYear(SaleDate);
</code></pre>

<p><strong>Note:</strong> Partition key (SaleDate) must be part of PRIMARY KEY!</p>
<h3 id="date-based-partitioning-example">Date-Based Partitioning Example</h3>
<p><strong>Monthly Partitioning for High-Volume Tables:</strong></p>
<pre class="codehilite"><code class="language-sql">-- Partition function for monthly partitions
CREATE PARTITION FUNCTION PF_TransactionsByMonth (DATETIME2)
AS RANGE RIGHT FOR VALUES
(
    '2024-01-01', '2024-02-01', '2024-03-01', '2024-04-01',
    '2024-05-01', '2024-06-01', '2024-07-01', '2024-08-01',
    '2024-09-01', '2024-10-01', '2024-11-01', '2024-12-01'
);

-- Partition scheme (all on PRIMARY for simplicity)
CREATE PARTITION SCHEME PS_TransactionsByMonth
AS PARTITION PF_TransactionsByMonth

-- Partitioned table

    TransactionType VARCHAR(50),

</code></pre>

<h2 id="partition-strategies">Partition Strategies</h2>
<h3 id="range-partitioning">Range Partitioning</h3>
<p>Most common strategy - partition by continuous ranges.</p>
<p><strong>Use Cases:</strong>
- Time-series data (orders, logs, events)
- Sequential IDs
- Date-based archiving</p>
<pre class="codehilite"><code class="language-sql">-- Example: Order history by year
CREATE PARTITION FUNCTION PF_OrdersByYear (DATE)
AS RANGE RIGHT FOR VALUES
    ('2020-01-01', '2021-01-01', '2022-01-01', '2023-01-01');
</code></pre>

<h3 id="list-partitioning">List Partitioning</h3>
<p>Partition by discrete values (region, category, status).</p>
<p><strong>Note:</strong> SQL Server doesn't have native LIST partitioning, but you can simulate it:</p>
<pre class="codehilite"><code class="language-sql">-- Simulate list partitioning by region
-- Use integer mapping: 1=North, 2=South, 3=East, 4=West

CREATE PARTITION FUNCTION PF_ByRegion (INT)
AS RANGE RIGHT FOR VALUES (2, 3, 4);

-- Partitions:
-- 1: RegionID = 1 (North)
-- 2: RegionID = 2 (South)
-- 3: RegionID = 3 (East)
-- 4: RegionID &gt;= 4 (West and others)
</code></pre>

<h3 id="hash-partitioning">Hash Partitioning</h3>
<p>Distribute data evenly across partitions using a hash function.</p>
<pre class="codehilite"><code class="language-sql">-- Simulated hash partitioning
-- Use computed column with hash function

CREATE TABLE Orders (
    OrderID BIGINT,
    CustomerID INT,
    OrderDate DATE,
    -- Computed column for partitioning
    PartitionKey AS (OrderID % 10) PERSISTED,
    CONSTRAINT PK_Orders PRIMARY KEY (OrderID, PartitionKey)
) ON PS_HashPartition(PartitionKey);

CREATE PARTITION FUNCTION PF_Hash (INT)
AS RANGE RIGHT FOR VALUES (1, 2, 3, 4, 5, 6, 7, 8, 9);
-- Creates 10 partitions (0-9)
</code></pre>

<h3 id="adding-new-partitions">Adding New Partitions</h3>
<p><strong>Method 1: Split Existing Partition</strong></p>
<pre class="codehilite"><code class="language-sql">-- Add new partition for 2025
ALTER PARTITION SCHEME PS_SalesByYear
NEXT USED FG_Sales_2025;  -- Optional: specify filegroup

ALTER PARTITION FUNCTION PF_SalesByYear()
SPLIT RANGE ('2025-01-01');

-- Result: Creates new partition for data &gt;= 2025-01-01
</code></pre>

<p><strong>Method 2: Pre-create Empty Partitions</strong></p>
<pre class="codehilite"><code class="language-sql">-- Create partition function with future boundaries
CREATE PARTITION FUNCTION PF_SalesWithFuture (DATE)
AS RANGE RIGHT FOR VALUES
(
    '2023-01-01', '2024-01-01', '2025-01-01',  -- Active
    '2026-01-01', '2027-01-01'                  -- Future (empty)
);
</code></pre>

<h3 id="removing-old-partitions">Removing Old Partitions</h3>
<p><strong>Method 1: Merge Empty Partition</strong></p>
<pre class="codehilite"><code class="language-sql">-- First, remove/archive data from oldest partition
-- Then merge the boundary

ALTER PARTITION FUNCTION PF_SalesByYear()
MERGE RANGE ('2020-01-01');

-- Result: Merges partition 1 and 2
-- Data that was &gt;= 2020 is now in same partition as &lt; 2020
</code></pre>

<p><strong>Method 2: Switch Out Before Merge (Recommended)</strong></p>
<pre class="codehilite"><code class="language-sql">-- Step 1: Create staging table with same structure
CREATE TABLE Sales_Archive_2020 (
    -- Same structure as Sales table
    SaleID BIGINT,
    SaleDate DATE NOT NULL,
    CustomerID INT,
    Amount DECIMAL(18,2),
    CONSTRAINT PK_Sales_Archive_2020 PRIMARY KEY (SaleID, SaleDate)
) ON FG_Sales_2020;  -- Same filegroup

-- Step 2: Switch partition to staging table
ALTER TABLE Sales
SWITCH PARTITION 1 TO Sales_Archive_2020;
-- Instant operation! No data movement

-- Step 3: Now merge the empty partition
ALTER PARTITION FUNCTION PF_SalesByYear()
MERGE RANGE ('2020-01-01');

-- Step 4: Archive or drop staging table
-- Backup Sales_Archive_2020 to tape/blob storage
-- DROP TABLE Sales_Archive_2020;
</code></pre>

<h3 id="switching-partitions">Switching Partitions</h3>
<p>Partition switching is the killer feature - instant data movement!</p>
<pre class="codehilite"><code class="language-sql">-- Switch partition 5 from Sales to Archive table
ALTER TABLE Sales
SWITCH PARTITION 5 TO Sales_Archive;

-- Requirements for SWITCH to work:
-- 1. Both tables same structure
-- 2. Both on same filegroup (for that partition)
-- 3. Constraints and indexes compatible
-- 4. No FK constraints pointing to table
</code></pre>

<p><strong>Complete Sliding Window Example:</strong></p>
<pre class="codehilite"><code class="language-sql">-- Monthly sliding window maintenance
-- Run at end of each month

DECLARE @NewMonth DATE = '2024-12-01';
DECLARE @OldMonth DATE = '2024-06-01';

-- Step 1: Add new partition for next month
ALTER PARTITION SCHEME PS_SalesByMonth
NEXT USED FG_Sales_Current;

ALTER PARTITION FUNCTION PF_SalesByMonth()
SPLIT RANGE (@NewMonth);

-- Step 2: Switch out old partition
CREATE TABLE Sales_Archive_202406 (
    SaleID BIGINT,
    SaleDate DATE NOT NULL,
    CustomerID INT,
    Amount DECIMAL(18,2),
    CONSTRAINT PK_Archive PRIMARY KEY (SaleID, SaleDate)
) ON FG_Sales_Archive;

ALTER TABLE Sales
SWITCH PARTITION $PARTITION.PF_SalesByMonth(@OldMonth)
TO Sales_Archive_202406;

-- Step 3: Merge old partition boundary
ALTER PARTITION FUNCTION PF_SalesByMonth()
MERGE RANGE (@OldMonth);

-- Step 4: Archive old data
BACKUP DATABASE SalesDB
FILEGROUP = 'FG_Sales_Archive'
TO DISK = 'E:\Backups\Sales_202406.bak';

DROP TABLE Sales_Archive_202406;
</code></pre>

<h2 id="performance-optimization">Performance Optimization</h2>
<h3 id="partition-elimination">Partition Elimination</h3>
<p>SQL Server automatically eliminates irrelevant partitions from query execution.</p>
<pre class="codehilite"><code class="language-sql">-- Query with partition elimination
SELECT *
FROM Sales
WHERE SaleDate &gt;= '2023-01-01'
  AND SaleDate &lt; '2023-02-01';

-- Execution plan shows:
-- Partitions scanned: 1 of 5
-- I/O reduced by 80%!
</code></pre>

<p><strong>Viewing Partition Elimination:</strong></p>
<pre class="codehilite"><code class="language-sql">-- Check actual execution plan
-- Look for &quot;Actual Partition Count&quot; vs &quot;Estimated Partition Count&quot;

-- Query DMVs for partition access
SELECT
    OBJECT_NAME(i.object_id) AS TableName,
    p.partition_number,
    p.rows,
    fg.name AS FileGroupName
FROM sys.partitions p
JOIN sys.indexes i ON p.object_id = i.object_id
    AND p.index_id = i.index_id
JOIN sys.allocation_units au ON p.partition_id = au.container_id
JOIN sys.filegroups fg ON au.data_space_id = fg.data_space_id
WHERE i.object_id = OBJECT_ID('Sales')
ORDER BY p.partition_number;
</code></pre>

<h3 id="aligned-indexes">Aligned Indexes</h3>
<p>Keep indexes <strong>aligned</strong> with table partitioning for best performance.</p>
<pre class="codehilite"><code class="language-sql">-- Aligned index (recommended)
CREATE NONCLUSTERED INDEX IX_Sales_Customer
ON Sales(CustomerID, SaleDate)
ON PS_SalesByYear(SaleDate);  -- Same partition scheme!

-- Non-aligned index (avoid if possible)
CREATE NONCLUSTERED INDEX IX_Sales_Customer_NonAligned
ON Sales(CustomerID)
ON [PRIMARY];  -- Different partition scheme
</code></pre>

<p><strong>Benefits of Aligned Indexes:</strong>
- âœ… Partition-level index maintenance
- âœ… Fast partition switching
- âœ… Better query performance
- âœ… Parallel operations</p>
<h3 id="partition-level-statistics">Partition-Level Statistics</h3>
<p>Update statistics per partition for faster maintenance:</p>
<pre class="codehilite"><code class="language-sql">-- Update statistics for specific partition
UPDATE STATISTICS Sales
WITH RESAMPLE
ON PARTITIONS (5);  -- Only partition 5

-- Rebuild index for specific partition
ALTER INDEX IX_Sales_Customer
ON Sales
REBUILD PARTITION = 5;

-- Benefits:
-- - Shorter maintenance window
-- - Less locking
-- - Can maintain partitions in parallel
</code></pre>

<h2 id="best-practices">Best Practices</h2>
<h3 id="choosing-partition-keys">Choosing Partition Keys</h3>
<p><strong>Good Partition Key Characteristics:</strong></p>
<pre class="codehilite"><code>âœ… Used in WHERE clauses frequently
âœ… Evenly distributed data
âœ… Stable (doesn't change)
âœ… Business-logical boundaries
âœ… Enables efficient archiving

âŒ Avoid skewed distributions
âŒ Avoid frequently updated columns
âŒ Avoid too many partitions (&lt; 1000)
âŒ Avoid too few partitions (&gt; 10-15)
</code></pre>

<p><strong>Examples:</strong></p>
<pre class="codehilite"><code class="language-sql">-- âœ… Good: Date column for time-series data
CREATE PARTITION FUNCTION PF_ByDate (DATE)
AS RANGE RIGHT FOR VALUES ('2023-01-01', '2024-01-01');

-- âœ… Good: Sequential ID with even distribution
CREATE PARTITION FUNCTION PF_ByCustomerID (INT)
AS RANGE RIGHT FOR VALUES (1000000, 2000000, 3000000);

-- âŒ Bad: Status column (uneven distribution)
CREATE PARTITION FUNCTION PF_ByStatus (VARCHAR(20))
AS RANGE RIGHT FOR VALUES ('Active', 'Inactive');
-- Problem: 99% data in 'Active', 1% in 'Inactive'

-- âŒ Bad: Updated column
CREATE PARTITION FUNCTION PF_ByLastModified (DATETIME)
AS RANGE RIGHT FOR VALUES (...);
-- Problem: Records move between partitions on update
</code></pre>

<h3 id="filegroup-strategy">Filegroup Strategy</h3>
<p><strong>Strategy 1: Separate Filegroups (Best for Large Databases)</strong></p>
<pre class="codehilite"><code class="language-sql">-- Each partition on different filegroup
-- Benefits:
-- - Backup/restore individual partitions
-- - Place on different disks for I/O distribution
-- - Better manageability

CREATE PARTITION SCHEME PS_Advanced
AS PARTITION PF_SalesByYear
TO (FG_2020, FG_2021, FG_2022, FG_2023, FG_2024);
</code></pre>

<p><strong>Strategy 2: All on PRIMARY (Simplest)</strong></p>
<pre class="codehilite"><code class="language-sql">-- All partitions on PRIMARY
-- Benefits:
-- - Simpler management
-- - Good for smaller databases
-- - Still get query performance benefits

CREATE PARTITION SCHEME PS_Simple
AS PARTITION PF_SalesByYear
ALL TO ([PRIMARY]);
</code></pre>

<p><strong>Strategy 3: Hot/Warm/Cold (Tiered Storage)</strong></p>
<pre class="codehilite"><code class="language-sql">-- Recent data on fast SSD, old data on slow HDD
CREATE PARTITION SCHEME PS_Tiered
AS PARTITION PF_SalesByYear
TO (
    FG_Cold_HDD,      -- 2020 data
    FG_Warm_HDD,      -- 2021 data
    FG_Warm_SSD,      -- 2022 data
    FG_Hot_SSD,       -- 2023 data
    FG_Hot_SSD        -- 2024+ data
);
</code></pre>

<h3 id="index-alignment">Index Alignment</h3>
<p><strong>Always align indexes with table partitioning:</strong></p>
<pre class="codehilite"><code class="language-sql">-- âœ… Aligned index (recommended)
CREATE INDEX IX_Aligned
ON Sales(CustomerID, SaleDate)
ON PS_SalesByYear(SaleDate);
-- Benefits: Partition-level maintenance, faster switching

-- âŒ Non-aligned (avoid)
CREATE INDEX IX_NonAligned
ON Sales(CustomerID)
ON [PRIMARY];
-- Problem: Can't switch partitions, slower maintenance
</code></pre>

<h3 id="monitoring-and-maintenance">Monitoring and Maintenance</h3>
<p><strong>Query Partition Information:</strong></p>
<pre class="codehilite"><code class="language-sql">-- View partition details
SELECT
    OBJECT_NAME(p.object_id) AS TableName,
    p.partition_number AS PartitionNum,
    prv.value AS BoundaryValue,
    p.rows AS RowCount,
    au.total_pages * 8 / 1024 AS SizeMB,
    fg.name AS FileGroupName
FROM sys.partitions p
LEFT JOIN sys.partition_range_values prv
    ON p.partition_id = prv.boundary_id
JOIN sys.allocation_units au
    ON p.partition_id = au.container_id
JOIN sys.filegroups fg
    ON au.data_space_id = fg.data_space_id
WHERE p.object_id = OBJECT_ID('Sales')
  AND p.index_id IN (0,1)  -- Heap or clustered index
ORDER BY p.partition_number;
</code></pre>

<p><strong>Monitor Partition Sizes:</strong></p>
<pre class="codehilite"><code class="language-sql">-- Alert if partition size exceeds threshold
WITH PartitionSizes AS (
    SELECT
        p.partition_number,
        SUM(au.total_pages) * 8 / 1024 AS SizeMB
    FROM sys.partitions p
    JOIN sys.allocation_units au
        ON p.partition_id = au.container_id
    WHERE p.object_id = OBJECT_ID('Sales')
    GROUP BY p.partition_number
)
SELECT *
FROM PartitionSizes
WHERE SizeMB &gt; 10000  -- Alert if &gt; 10GB
ORDER BY partition_number;
</code></pre>

<h2 id="common-scenarios">Common Scenarios</h2>
<h3 id="time-based-data-archiving">Time-Based Data Archiving</h3>
<p><strong>Scenario:</strong> Retain 2 years of online data, archive older data</p>
<pre class="codehilite"><code class="language-sql">-- Initial setup
CREATE PARTITION FUNCTION PF_OrderArchive (DATE)
AS RANGE RIGHT FOR VALUES
(
    '2022-01-01',  -- Old data
    '2023-01-01',  -- Last year
    '2024-01-01'   -- Current year
);

CREATE PARTITION SCHEME PS_OrderArchive
AS PARTITION PF_OrderArchive
TO (FG_Archive, FG_History, FG_Current, FG_Current);

-- Monthly maintenance job
-- Add new month, archive old month
DECLARE @NewMonth DATE = DATEADD(MONTH, 1, EOMONTH(GETDATE()));
DECLARE @ArchiveMonth DATE = DATEADD(YEAR, -2, DATEADD(MONTH, 1, EOMONTH(GETDATE())));

-- Add new partition
ALTER PARTITION SCHEME PS_OrderArchive NEXT USED FG_Current;
ALTER PARTITION FUNCTION PF_OrderArchive() SPLIT RANGE (@NewMonth);

-- Archive old partition
-- Switch to archive table, backup, merge
</code></pre>

<h3 id="multi-tenant-applications">Multi-Tenant Applications</h3>
<p><strong>Scenario:</strong> Separate data by tenant ID for isolation and performance</p>
<pre class="codehilite"><code class="language-sql">-- Partition by TenantID ranges
CREATE PARTITION FUNCTION PF_ByTenant (INT)
AS RANGE RIGHT FOR VALUES
(
    1000,  -- Tenants 1-999
    2000,  -- Tenants 1000-1999
    3000,  -- Tenants 2000-2999
    4000   -- Tenants 3000-3999
);

CREATE TABLE CustomerData (
    DataID BIGINT IDENTITY,
    TenantID INT NOT NULL,
    CustomerName NVARCHAR(200),
    Data XML,
    CONSTRAINT PK_Data PRIMARY KEY (DataID, TenantID)
) ON PS_ByTenant(TenantID);

-- Benefits:
-- - Tenant data isolation
-- - Backup/restore per tenant
-- - Easier tenant migration
-- - Better query performance with partition elimination
</code></pre>

<h3 id="geographic-data-distribution">Geographic Data Distribution</h3>
<p><strong>Scenario:</strong> Partition by region for multi-region application</p>
<pre class="codehilite"><code class="language-sql">-- Map regions to partition numbers
-- 1=US, 2=EU, 3=APAC, 4=Other

CREATE PARTITION FUNCTION PF_ByRegion (TINYINT)
AS RANGE RIGHT FOR VALUES (2, 3, 4);

CREATE PARTITION SCHEME PS_ByRegion
AS PARTITION PF_ByRegion
TO (FG_US, FG_EU, FG_APAC, FG_Other);

CREATE TABLE Orders (
    OrderID BIGINT,
    RegionID TINYINT NOT NULL,
    OrderDate DATE,
    CustomerID INT,
    CONSTRAINT PK_Orders PRIMARY KEY (OrderID, RegionID)
) ON PS_ByRegion(RegionID);

-- Can place filegroups on servers in respective regions
-- for better performance
</code></pre>

<h2 id="troubleshooting">Troubleshooting</h2>
<h3 id="performance-issues">Performance Issues</h3>
<p><strong>Problem: Queries still slow after partitioning</strong></p>
<p><strong>Diagnosis:</strong></p>
<pre class="codehilite"><code class="language-sql">-- Check if partition elimination is working
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT *
FROM Sales
WHERE SaleDate = '2023-05-15';

-- Look for:
-- - &quot;Partitions accessed: 1&quot; (good)
-- - &quot;Partitions accessed: all&quot; (bad - no elimination)
</code></pre>

<p><strong>Solutions:</strong></p>
<pre class="codehilite"><code class="language-sql">-- âœ… Include partition key in WHERE clause
SELECT * FROM Sales
WHERE SaleDate &gt;= '2023-01-01';  -- Good

-- âŒ Avoid functions on partition key
SELECT * FROM Sales
WHERE YEAR(SaleDate) = 2023;  -- Bad - no elimination

-- âœ… Use SARGable predicates
SELECT * FROM Sales
WHERE SaleDate &gt;= '2023-01-01'
  AND SaleDate &lt; '2024-01-01';  -- Good
</code></pre>

<h3 id="partition-imbalance">Partition Imbalance</h3>
<p><strong>Problem: One partition much larger than others</strong></p>
<p><strong>Diagnosis:</strong></p>
<pre class="codehilite"><code class="language-sql">-- Find partition sizes
SELECT
    p.partition_number,
    p.rows,
    au.total_pages * 8 / 1024 AS SizeMB,
    CAST(100.0 * p.rows / SUM(p.rows) OVER() AS DECIMAL(5,2)) AS PctOfTotal
FROM sys.partitions p
JOIN sys.allocation_units au ON p.partition_id = au.container_id
WHERE p.object_id = OBJECT_ID('Sales')
  AND p.index_id IN (0,1)
ORDER BY p.rows DESC;
</code></pre>

<p><strong>Solutions:</strong></p>
<ul>
<li>Adjust partition boundaries</li>
<li>Split large partitions</li>
<li>Consider different partition key</li>
<li>Archive historical data</li>
</ul>
<h3 id="lock-contention">Lock Contention</h3>
<p><strong>Problem: Partition switch operations blocked</strong></p>
<p><strong>Diagnosis:</strong></p>
<pre class="codehilite"><code class="language-sql">-- Check for blocking
SELECT
    blocking.session_id AS BlockingSessionId,
    blocked.session_id AS BlockedSessionId,
    blocked.wait_type,
    blocked.wait_time,
    blocked.wait_resource
FROM sys.dm_exec_requests blocked
JOIN sys.dm_exec_requests blocking
    ON blocked.blocking_session_id = blocking.session_id
WHERE blocked.blocking_session_id &gt; 0;
</code></pre>

<p><strong>Solutions:</strong></p>
<pre class="codehilite"><code class="language-sql">-- Ensure no active transactions on table
-- Disable foreign keys temporarily
ALTER TABLE Orders NOCHECK CONSTRAINT ALL;

-- Perform switch
ALTER TABLE Sales SWITCH PARTITION 5 TO Sales_Archive;

-- Re-enable constraints
ALTER TABLE Orders CHECK CONSTRAINT ALL;
</code></pre>

<h2 id="conclusion">Conclusion</h2>
<p>SQL Server table partitioning is a powerful feature for managing large tables, improving query performance, and simplifying maintenance operations. By dividing massive tables into smaller, manageable partitions, you can:</p>
<h3 id="key-benefits-recap">Key Benefits Recap</h3>
<p><strong>Performance:</strong>
- Partition elimination dramatically reduces I/O
- Parallel operations on multiple partitions
- Faster index maintenance per partition</p>
<p><strong>Manageability:</strong>
- Instant data archiving with partition switching
- Partition-level backup and restore
- Sliding window pattern for automated maintenance</p>
<p><strong>Scalability:</strong>
- Handle billions of rows efficiently
- Linear scalability with partition count
- Better resource utilization</p>
<h3 id="implementation-guidelines">Implementation Guidelines</h3>
<ol>
<li><strong>Start Simple:</strong> Begin with yearly or monthly partitions</li>
<li><strong>Measure First:</strong> Benchmark before and after partitioning</li>
<li><strong>Align Indexes:</strong> Keep indexes aligned with table partitioning</li>
<li><strong>Monitor Regularly:</strong> Track partition sizes and query performance</li>
<li><strong>Automate Maintenance:</strong> Use jobs for sliding window operations</li>
<li><strong>Plan Filegroups:</strong> Consider separate filegroups for flexibility</li>
</ol>
<h3 id="when-to-use-partitioning">When to Use Partitioning</h3>
<p><strong>Good Candidates:</strong>
- âœ… Tables &gt; 100 million rows
- âœ… Historical/time-series data
- âœ… Regular archiving requirements
- âœ… Known query patterns on partition key</p>
<p><strong>Not Recommended:</strong>
- âŒ Small tables (&lt; 10 million rows)
- âŒ Frequently updated partition key
- âŒ No clear partition boundary
- âŒ Random access patterns</p>
<p>Remember: Table partitioning requires <strong>SQL Server Enterprise Edition</strong> (or Azure SQL Database). The benefits often justify the cost for large-scale systems, but evaluate your specific needs and workload patterns.</p>
