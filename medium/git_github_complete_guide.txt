<h2 id="introduction">Introduction</h2>
<p>Imagine working on a document and accidentally deleting an important paragraph. Or imagine collaborating with teammates on the same file and losing track of who changed what. Now imagine having a time machine that records every change, lets you travel back to any previous version, and seamlessly merges everyone's work together. That's exactly what Git and GitHub provide for software development.</p>
<p>Git has revolutionized how developers work, transforming from chaotic file copying and naming conventions (like <code>final_version_v2_REALLY_FINAL.txt</code>) to an organized, powerful system that tracks every change, enables fearless experimentation, and facilitates seamless collaboration.</p>
<hr />
<h2 id="understanding-version-control">Understanding Version Control</h2>
<h3 id="why-version-control-matters">Why Version Control Matters</h3>
<p>Version control solves fundamental problems in software development:</p>
<p><strong>Problem 1: Change Tracking</strong>
- âŒ Without version control: Which line changed? When? Why?
- âœ… With version control: Complete history of every change with reasons</p>
<p><strong>Problem 2: Collaboration</strong>
- âŒ Without version control: Overwriting each other's changes, email attachments, confusion
- âœ… With version control: Parallel work with automatic merging</p>
<p><strong>Problem 3: Backup and Recovery</strong>
- âŒ Without version control: Lost work, manual backups, no rollback
- âœ… With version control: Every version saved, easy rollback to any point</p>
<p><strong>Problem 4: Experimentation</strong>
- âŒ Without version control: Fear of breaking working code
- âœ… With version control: Create branches, experiment freely, discard or merge</p>
<h3 id="real-world-analogy">Real-World Analogy</h3>
<p>Think of Git like a sophisticated <strong>"save game" system in video games</strong>, but for code:</p>
<ul>
<li><strong>Saves (Commits):</strong> Just like saving your progress in a game, commits save your code's state</li>
<li><strong>Multiple Save Slots (Branches):</strong> Try different strategies without losing your main progress</li>
<li><strong>Load Previous Save (Checkout):</strong> Travel back to any previous state</li>
<li><strong>Merge Strategies:</strong> Combine successful experiments into your main game</li>
<li><strong>Multiplayer (GitHub):</strong> Share your save files and collaborate with other players</li>
</ul>

<h3 id="core-git-concepts">Core Git Concepts</h3>
<p><strong>1. Repository (Repo)</strong>
A repository is a directory containing your project files and the complete history of all changes.</p>
<pre class="codehilite"><code class="language-bash"># A repository contains:

</code></pre>

<p><strong>2. Commit</strong>
A commit is a snapshot of your project at a specific point in time. Think of it as a save point.</p>
<pre class="codehilite"><code>Each commit contains:
- Snapshot of all files
- Author information
- Timestamp
- Commit message (description)
- Parent commit reference
- Unique SHA-1 hash (ID)
</code></pre>

<p><strong>3. Branch</strong>
A branch is an independent line of development. The default branch is usually called <code>main</code> or <code>master</code>.</p>
<pre class="codehilite"><code>        main branch
    A --- B --- C --- D
               \
                E --- F (feature branch)
</code></pre>

<p><strong>4. Remote</strong>
A remote is a version of your repository hosted elsewhere (like on GitHub).</p>

<h2 id="essential-git-commands">Essential Git Commands</h2>
<h3 id="repository-initialization">Repository Initialization</h3>
<p><strong>Creating a New Repository:</strong></p>
<pre class="codehilite"><code class="language-bash"># Start a new Git repository
git init

# This creates a .git directory containing all Git metadata
</code></pre>

<p><strong>Cloning an Existing Repository:</strong></p>
<pre class="codehilite"><code class="language-bash"># Clone from GitHub or other remote
git clone https://github.com/username/repository.git

# Clone with a custom directory name
git clone https://github.com/username/repository.git my-project
</code></pre>

<p><strong>Checking Repository Status:</strong></p>
<pre class="codehilite"><code class="language-bash"># See which files are modified, staged, or untracked
git status

# Shorter status output
git status -s
</code></pre>

<h3 id="branching-and-merging">Branching and Merging</h3>
<p><strong>Branch Management:</strong></p>
<pre class="codehilite"><code class="language-bash"># List all branches
git branch

# Create new branch
git branch feature-login

# Switch to branch
git checkout feature-login

# Create and switch in one command
git checkout -b feature-login

# Modern way (Git 2.23+)
git switch feature-login
git switch -c feature-login

# Delete branch
git branch -d feature-login

# Force delete unmerged branch
git branch -D feature-login

# Rename current branch
git branch -m new-name
</code></pre>

<p><strong>Merging:</strong></p>
<pre class="codehilite"><code class="language-bash"># Merge feature branch into current branch
git checkout main
git merge feature-login

# Merge with commit message
git merge feature-login -m &quot;Integrate login feature&quot;

# Abort merge if conflicts
git merge --abort
</code></pre>

<h3 id="remote-operations">Remote Operations</h3>
<p><strong>Working with Remotes:</strong></p>
<pre class="codehilite"><code class="language-bash"># View remote repositories
git remote -v

# Add a remote
git remote add origin https://github.com/username/repo.git

# Change remote URL
git remote set-url origin https://github.com/username/new-repo.git

# Remove remote
git remote remove origin

# Rename remote
git remote rename origin upstream
</code></pre>

<p><strong>Pushing and Pulling:</strong></p>
<pre class="codehilite"><code class="language-bash"># Push to remote
git push origin main

# Push and set upstream tracking
git push -u origin main

# Push all branches
git push --all

# Force push (DANGEROUS!)
git push --force

# Safer force push
git push --force-with-lease

# Pull changes from remote
git pull origin main

# Pull with rebase instead of merge
git pull --rebase

# Fetch changes without merging
git fetch origin

# View remote branches
git branch -r
</code></pre>

<h2 id="best-practices">Best Practices</h2>
<h3 id="commit-messages">Commit Messages</h3>
<p><strong>Anatomy of a Good Commit Message:</strong></p>
<pre class="codehilite"><code>Short summary (50 chars or less)

Detailed explanation if needed. Wrap at 72 characters.
Explain WHAT changed and WHY, not HOW (code shows how).

- Bullet points are okay
- Use imperative mood: &quot;Add feature&quot; not &quot;Added feature&quot;
- Reference issues: Fixes #123, Relates to #456
</code></pre>

<p><strong>Examples:</strong></p>
<pre class="codehilite"><code>âœ… Good:
Add user authentication with JWT tokens

Implement JWT-based authentication to secure API endpoints.
Users can now log in and receive a token that expires after
24 hours. This replaces the old session-based auth which was
causing scaling issues.

Fixes #234
</code></pre>

<pre class="codehilite"><code>âŒ Bad:
fixed stuff

updated some files and changed things
</code></pre>

<p><strong>Conventional Commits Format:</strong></p>
<pre class="codehilite"><code>type(scope): subject

feat(auth): add JWT token authentication
fix(ui): correct button alignment on mobile
docs(readme): update installation instructions
style(css): format code according to style guide
refactor(api): simplify error handling logic
test(auth): add unit tests for login flow
chore(deps): update dependencies to latest versions
</code></pre>

<h2 id="common-scenarios-and-solutions">Common Scenarios and Solutions</h2>
<h3 id="undoing-changes">Undoing Changes</h3>
<p><strong>Scenario 1: Undo Unstaged Changes</strong></p>
<pre class="codehilite"><code class="language-bash"># Discard changes in working directory
git restore file.txt

# Or older syntax
git checkout -- file.txt

# Discard all changes
git restore .
</code></pre>

<p><strong>Scenario 2: Unstage Files</strong></p>
<pre class="codehilite"><code class="language-bash"># Unstage specific file (keep changes)
git restore --staged file.txt

# Or older syntax
git reset HEAD file.txt

# Unstage all files
git restore --staged .
</code></pre>

<p><strong>Scenario 3: Undo Last Commit (Keep Changes)</strong></p>
<pre class="codehilite"><code class="language-bash"># Undo commit, keep changes staged
git reset --soft HEAD~1

# Undo commit, keep changes unstaged
git reset HEAD~1

# Undo commit, discard changes
git reset --hard HEAD~1
</code></pre>

<p><strong>Scenario 4: Amend Last Commit</strong></p>
<pre class="codehilite"><code class="language-bash"># Change last commit message
git commit --amend -m &quot;New message&quot;

# Add forgotten files to last commit
git add forgotten-file.txt
git commit --amend --no-edit
</code></pre>

<p><strong>Scenario 5: Revert a Commit (Safe for Public Branches)</strong></p>
<pre class="codehilite"><code class="language-bash"># Create new commit that undoes changes
git revert commit-hash

# Revert without committing (review first)
git revert --no-commit commit-hash
</code></pre>

<h3 id="stashing-work">Stashing Work</h3>
<p>Save work temporarily without committing:</p>
<pre class="codehilite"><code class="language-bash"># Stash current changes
git stash

# Stash with description
git stash save &quot;Work in progress on login feature&quot;

# List stashes
git stash list

# Apply most recent stash
git stash apply

# Apply and remove stash
git stash pop

# Apply specific stash
git stash apply stash@{2}

# Delete specific stash
git stash drop stash@{0}

# Clear all stashes
git stash clear

# Stash including untracked files
git stash -u
</code></pre>

<p><strong>Use Case:</strong></p>
<pre class="codehilite"><code class="language-bash"># Working on feature, need to fix urgent bug
git stash                   # Save current work
git checkout main           # Switch to main
git checkout -b hotfix-123  # Create hotfix branch
# ... fix bug ...
git commit -m &quot;Fix critical bug&quot;
git checkout feature-branch # Back to feature
git stash pop              # Resume work
</code></pre>

<h3 id="cherry-picking-commits">Cherry-Picking Commits</h3>
<p>Apply specific commits from one branch to another:</p>
<pre class="codehilite"><code class="language-bash"># Apply specific commit to current branch
git cherry-pick commit-hash

# Cherry-pick multiple commits
git cherry-pick commit1 commit2 commit3

# Cherry-pick without committing
git cherry-pick --no-commit commit-hash
</code></pre>

<p><strong>Visual Example:</strong></p>
<pre class="codehilite"><code>main:     A --- B --- C --- E
                   \
feature:            D --- F --- G

# Want commit F on main
git checkout main
git cherry-pick F

main:     A --- B --- C --- E --- F'
                   \
feature:            D --- F --- G

# F' is a copy of F
</code></pre>

<h3 id="submodules">Submodules</h3>

<h3 id="key-takeaways-recap">Key Takeaways Recap</h3>
<p><strong>Git Fundamentals:</strong>
- Git provides complete version history and change tracking
- The three states (working, staging, committed) give precise control
- Branches enable parallel development without interference</p>
<p><strong>Collaboration:</strong>
- GitHub extends Git with powerful collaboration features
- Pull requests facilitate code review and team coordination
- Clear communication and good practices prevent conflicts</p>
<p><strong>Best Practices:</strong>
- Write clear, descriptive commit messages
- Use meaningful branch names
- Keep commits small and focused
- Review code thoughtfully
- Follow team conventions</p>
<p><strong>Problem Solving:</strong>
- Git provides tools to undo mistakes safely
- Conflicts are normal and manageable
- Stashing helps manage context switching
- Understanding your workflow prevents issues</p>
